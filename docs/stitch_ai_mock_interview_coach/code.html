好的。核心功能需求主要有两个：一是将长 URL 转换为一个较短的 URL；二是当用户访问短 URL 时，系统能重定向到原始的长 URL。

非功能需求方面，系统需要具备高可用性，因为重定向服务不能宕机；另外响应延迟要极低；生成的短链接应该是不可预测的，以防止被恶意遍历。

smart_toy

很好的总结。你提到了高可用性和低延迟。

考虑到这是一个读多写少的系统（重定向请求远大于生成请求），针对“高可用”和“低延迟”这两个非功能需求，在架构设计上你会采取哪些具体的策略？另外，如果我们需要支持自定义短链接后缀，会对系统设计产生什么影响？

person

为了保证低延迟，我会在系统中引入缓存层，比如 Redis，将频繁访问的短链接映射缓存在内存中。对于高可用，可以通过部署多个应用节点配合负载均衡器来实现无状态服务，数据库也可以采用主从架构，读写分离来提高读取性能。

关于自定义后缀，这意味着我们需要在生成链接时检查全局唯一性，这会引入额外的数据库查询，可能成为性能瓶颈，并且由于可能产生冲突，需要设计冲突解决机制。
